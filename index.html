02

What Is an App, Really?

An app is just files in a folder. That's it. When you open a website, your browser reads these text files and turns them into something visual. That's all that's happening.

my-app/
├── index.html
├── styles.css
└── script.js
This presentation you're looking at right now? It's a single HTML file. You can open it in any text editor and read it.
03

The Terminal

The terminal is just another way to talk to your computer — by typing instead of clicking. You don't need to master it. You just need to recognize what Claude Code is doing when you see terminal commands flash by.

Terminal
$ ls→ list files in current folder
$ cd my-app→ move into a folder
$ open .→ open current folder in Finder

Claude Code lives in the terminal. You type claude and press enter. That's it.

04

Git: Your Safety Net

Git is a save system for your project. Like save points in a video game.

Initial commit
Add header
Style buttons
You are here
Next change
  1. 1 Commits = snapshots of your project at a point in time
  2. 2 Branches = parallel “what if” explorations that don't affect your main project
  3. 3 You can always go back — nothing is permanent, nothing is scary
Claude Code handles git for you. You just need to understand the concept.

GitHub is where your project lives online. It's like Google Drive for code.

The New Employee Problem

Every Claude Code session is like a brilliant but amnesiac contractor showing up to work.

Imagine hiring the world's best developer, but every morning they wake up with zero memory of your project.

What would you do?

You'd write everything down.

That's exactly what CLAUDE.md is for →

06

CLAUDE.md — Your Onboarding Doc

This is the document Claude Code reads at the start of every session. It's your project's institutional memory.

# Project: My App ## What This Is A mobile app for tracking daily habits. ## Tech Stack - React Native + Expo - TypeScript ## Architecture - /src/components — UI components - /src/screens — App screens ## Key Decisions - Tab-based navigation - All colors in theme.ts - Never use inline styles
Think of it like a design system doc, but for your entire project. The better this document is, the better Claude Code performs.

Keep tasks focused and discrete. Don't say “rebuild my entire app.” Say “update the header component to match this new design.”
08

Plan Mode

You can ask Claude Code to think before it acts.

Plan mode is like asking a developer to write a technical spec before they start coding. Claude Code will outline what it's going to do, what files it'll touch, and what approach it'll take.

Terminal
$ claude
> shift+tab to enter plan mode
> Plan how to add dark mode to the app

When to use it: Anything non-trivial — new features, refactoring, or when you're not sure how to approach a problem.

You wouldn't jump straight into high-fidelity mocks without exploring the problem first. Same idea.
09

Prompt as Design Spec

The quality of Claude Code's output is directly proportional to the quality of your prompt. Designers are already good at this — you just need to apply it differently.

Vague prompt

“Make a login page”

Design-quality prompt

“Create a login page with an email and password field, a primary 'Sign In' button using our brand blue (#2563eb), a 'Forgot password?' link below, and a 'Create account' link at the bottom. Use 16px body text, 24px heading. Center everything vertically with max-width 400px.”

You already write specs like this for developers. Now you're just writing them for a different kind of developer.

Figma MCP

Claude Code can look directly at your Figma designs.

MCP is a plugin system that lets Claude Code connect to other tools. The Figma MCP lets Claude Code read your Figma files — layers, styles, layout, components.

You can say “look at this Figma frame and build it” and Claude Code will reference the actual design, not just your description of it.

This is where your existing Figma workflow connects directly to code output.
11

Reading Code
(You Already Can)

You don't need to write code. You need to be able to scan it enough to verify Claude Code did what you asked.

<div class="card"> ← a container (like a frame in Figma) <h2>Title</h2> ← a heading <p>Description</p> ← body text <button>Click me</button> ← a button </div>
.card { padding: 24px; ← inner spacing border-radius: 12px; ← corner radius background: #ffffff; ← fill color box-shadow: 0 2px 8px ← drop shadow rgba(0,0,0,0.1); }
If you've ever inspected an element in Figma's dev mode, you can read this. The vocabulary is almost identical.
12

The Feedback Loop

Working with Claude Code is an iterative conversation, just like a design critique.

  1. • Be specific: “The spacing between the cards is too tight” not “it looks off”
  2. • Reference visual properties: “Make the heading 20px instead of 16px”
  3. • If the conversation gets long and CC seems confused, start a new session. It's like a fresh page.

The Gap Is Closing

You don't need to become a developer. You need to become a designer who can speak to a very fast, very capable one.

Getting Started

Do this today:

  • Install Claude Code
  • Open your terminal
  • Navigate to an empty folder
  • Type claude and press enter
  • Say: “Create a simple webpage with a centered heading that says Hello World, using a dark background and white text”
  • Open the file in your browser
  • You just shipped code.
